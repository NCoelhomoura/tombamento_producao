TARGET:
O Objetivo desse documento é garantir que as regras de negócio e de migração dos dados seja seguida rigorosamente para que os dados cheguem ao destino sem a necessidade de serem tratados no novo sistema 


IMPORTANTE - REGRAS DE LIMPEZA DE DADOS (TRUNCATE vs DELETE):

⚠️ ATENÇÃO: Quando filtros são aplicados (IdOrcamento, DataAvisoPrevio, DataInicioOperacao), NÃO pode ser usado TRUNCATE a menos que tenha um comando master como "--clear-data".

REGRAS GERAIS:
1. SEM filtros aplicados E SEM flag --clear-data:
   - Usar TRUNCATE (apaga todos os dados da tabela)
   - Comportamento padrão quando migrando todos os dados

2. COM filtros aplicados E SEM flag --clear-data:
   - NÃO usar TRUNCATE (apagaria dados não migrados nesta execução)
   - Usar DELETE com WHERE baseado nos filtros aplicados
   - Exemplo: DELETE FROM contracts WHERE legacy_id IN (lista de IdOrcamento dos filtros aplicados)

3. SEM filtros aplicados E COM flag --clear-data:
   - Usar TRUNCATE (mesmo comportamento do caso 1, mas explícito)

4. COM filtros aplicados E COM flag --clear-data:
   - Usar TRUNCATE (ignora filtros para limpeza, limpa tudo)
   - ⚠️ IMPORTANTE: Limpa COMPLETAMENTE a tabela antes de migrar apenas os dados filtrados

LÓGICA DE DECISÃO (aplicada em ordem):
1. Se clear_data=True → SEMPRE TRUNCATE (mesmo com filtros)
2. Se clear_data=False E há filtros → DELETE (apenas registros filtrados)
3. Se clear_data=False E NÃO há filtros → TRUNCATE (comportamento padrão)

EXEMPLO DE DETECÇÃO DE FILTROS:
- Se IdOrcamento foi especificado: há filtro → usar DELETE (se clear_data=False)
- Se DataAvisoPrevio foi especificado: há filtro → usar DELETE (se clear_data=False)
- Se DataInicioOperacao foi especificado: há filtro → usar DELETE (se clear_data=False)
- Se nenhum filtro foi aplicado: usar TRUNCATE (comportamento padrão)
- Se clear_data=True: SEMPRE TRUNCATE (ignora qualquer filtro)


-- FLAG --clear-data: REGRAS ESPECÍFICAS --

⚠️ CRÍTICO: O flag --clear-data NÃO pode apagar dados de tabelas que NÃO serão atualizadas na execução atual.

REGRAS DO --clear-data:
1. Escopo limitado: O --clear-data só pode limpar (TRUNCATE) tabelas que serão migradas na execução atual
   - Se executar apenas step1: pode limpar apenas contracts
   - Se executar step1 e step2: pode limpar contracts e contract_scenarios
   - NUNCA limpar tabelas de steps que não serão executados

2. Comportamento com filtros:
   - SEM filtros + --clear-data: TRUNCATE nas tabelas do escopo atual
   - COM filtros + --clear-data: TRUNCATE nas tabelas do escopo atual (ignora filtros para limpeza)
   - ⚠️ CRÍTICO: Quando clear_data=True, SEMPRE usa TRUNCATE, mesmo quando há filtros aplicados
   - Use apenas quando tiver certeza que deseja limpar COMPLETAMENTE as tabelas do escopo antes de migrar
   
3. Lógica implementada no código:
   - if clear_data=True: → TRUNCATE (sempre, mesmo com filtros)
   - elif há filtros E clear_data=False: → DELETE (apenas registros filtrados)
   - elif NÃO há filtros E clear_data=False: → TRUNCATE (comportamento padrão)

3. Exemplos de uso seguro:
   ✅ python orchestrator_tasks.py contracts 1 --clear-data
      → Limpa apenas contracts (única tabela do step1)
   
   ✅ python orchestrator_tasks.py contracts --clear-data
      → Limpa contracts, contract_scenarios, contract_scenario_stores, contract_sellers, 
        contract_team_members, contract_contacts, contract_partners, contract_additional_charges
        (todas as tabelas que serão migradas)
   
   ❌ python orchestrator_tasks.py contracts 1 --clear-data
      → NÃO pode limpar contract_scenarios, contract_sellers, etc. (steps não executados)

4. Quando usar --clear-data:
   - Migração completa de um módulo (sem filtros)
   - Reset completo antes de nova migração
   - Testes em ambiente de desenvolvimento
   - Quando precisa garantir que não há dados antigos misturados

5. Quando NÃO usar --clear-data:
   - Migração incremental com filtros
   - Quando outros steps já foram executados anteriormente
   - Em produção sem backup adequado
   - Quando não tem certeza do escopo da execução


ORIGIN RULES

-- CONTRACTS --

-- STEP 1: step1_migrate_contracts

Tabela Origem: ViewOrcamentosLojas + Orcamento (INNER JOIN)
Tabela Destino: contracts

LIMPEZA DE DADOS:
- SEM filtros aplicados: TRUNCATE TABLE contracts CASCADE
- COM filtros aplicados: DELETE FROM contracts WHERE legacy_id IN (lista de IdOrcamento dos filtros)
- COM flag --clear-data: TRUNCATE TABLE contracts CASCADE (mesmo com filtros)

Pegar todos os contratos da ViewOrcamentosLojas seguindo os filtros abaixo:
Todos os filtros são OPCIONAIS e devem ser aplicados apenas se definidos via parâmetros de linha de comando ou configuração.

- IdOrcamento: Não é obrigatório, mas filtrar caso tiver definido um ou mais orçamentos específicos e separar esses orçamentos por vírgula
Exemplo de uso: IdOrcamento IN (valores definidos via parâmetros de linha de comando ou configuração)

- DataAvisoPrevio: Não é obrigatório, mas filtrar caso tiver definido uma data mínima
Exemplo de uso: 
AND (
		CONVERT(DATE, DataAvisoPrevio) >= '2025-01-01'
		OR DataAvisoPrevio IS NULL
	)

- DataInicioOperacao: Não é obrigatório, mas filtrar caso tiver definido uma data máxima
Exemplo de uso: 
	AND CONVERT(DATE, DataInicioOperacao) <= '2026-01-13'


* USE CASE *
Exemplo de script filtrado seguindo modelo que já está definido no projeto inicial:
SELECT 
    v.IdOrcamento,
    MAX(v.IdCliente) AS IdCliente,
    MAX(v.DiaFaturamento) AS DiaFaturamento,
    MAX(v.DiaVencimento) AS DiaVencimento,
    MAX(v.StatusPedido) AS StatusPedido,
    MAX(v.DataInicioOperacao) AS DataInicioOperacao,
    MAX(v.DataInclusaoOrcamento) AS DataInclusaoOrcamento,
    MAX(v.DataAlteracaoOrcamento) AS DataAlteracaoOrcamento,
    MAX(o.ModoFaturamento) AS ModoFaturamento,
    MAX(o.TipoOrcamento) AS TipoOrcamento,
    MAX(o.TipoCalculoDecimoTerceiro) AS TipoCalculoDecimoTerceiro,
    MAX(o.TradeMarketing) AS TradeMarketing
FROM ViewOrcamentosLojas v
    INNER JOIN Orcamento o ON o.Id = v.IdOrcamento
WHERE IdOrcamento IN (valores do filtro IdOrcamento se aplicado)
        AND (
            CONVERT(DATE, DataAvisoPrevio) >= (data mínima do filtro DataAvisoPrevio se aplicado)
            OR DataAvisoPrevio IS NULL
        )
        AND CONVERT(DATE, DataInicioOperacao) <= (data máxima do filtro DataInicioOperacao se aplicado)
GROUP BY v.IdOrcamento
ORDER BY v.IdOrcamento

Após a execução do step1_migrate_contracts, será gerado automaticamente um arquivo JSON chamado "contracts_filter_main.json" na pasta contracts/ contendo:
- Lista de IdOrcamento migrados (para uso como filtro nos steps seguintes)
- Lista agregada de IdCliente, IdEstabelecimento, IdBandeira, IdRede (para referência)
- Filtros aplicados (para rastreabilidade e debug)


-- STEPS DENTRO DE CONTRACTS --

Dentro de Contracts existem vários STEPS (2 a 8) e todos devem seguir a lógica da step1_migrate_contracts.
A lógica é pegar tudo que pertence ao step1_migrate_contracts, ou seja, pegar o resultado de step1_migrate_contracts e usar esse resultado como filtro.

Os steps 2 a 8 devem usar o arquivo JSON gerado (contracts/contracts_filter_main.json) para filtrar seus dados, garantindo que apenas dados relacionados aos contratos migrados no step1 sejam processados.


-- STEP 9: step9_migrate_promoter_tasks

⚠️ CRÍTICO: Esta etapa DEVE ser executada ANTES do step2_migrate_contract_scenarios, pois contract_scenarios depende de promoter_tasks estar migrado.

Tabela Origem: ViewOrcamentosLojas (IdTarefa, NomeTarefa) + dbo.Tarefa (LEFT JOIN)
Tabela Destino: pdv.promoter_tasks

LIMPEZA DE DADOS:
⚠️ ATENÇÃO: promoter_tasks é uma tabela de referência compartilhada. NÃO usar TRUNCATE ou DELETE com filtros.
- SEMPRE usar UPSERT (INSERT ... ON CONFLICT ... DO UPDATE) baseado no campo `name`
- NÃO limpar dados existentes (tabela compartilhada, pode ser usada por outros módulos)
- Apenas inserir/atualizar tarefas encontradas na ViewOrcamentosLojas dos contratos filtrados

FILTRO OBRIGATÓRIO: Coletar tarefas únicas da ViewOrcamentosLojas filtrada pelo step1
Fonte do filtro: contracts/contracts_filter_main.json -> aggregated_ids.IdOrcamento

LÓGICA DE COLETA:
1. Coletar IdTarefa e NomeTarefa únicos da ViewOrcamentosLojas filtrada:
   SELECT DISTINCT
       v.IdTarefa,
       v.NomeTarefa
   FROM ViewOrcamentosLojas v
   INNER JOIN Orcamento o ON o.Id = v.IdOrcamento
   WHERE v.IdOrcamento IN (valores de contracts/contracts_filter_main.json -> aggregated_ids.IdOrcamento)
       AND (v.IdTarefa IS NOT NULL OR v.NomeTarefa IS NOT NULL)

2. Para cada tarefa única, buscar dados complementares de dbo.Tarefa:
   SELECT 
       t.Id,
       t.Nome,
       t.Ativo,
       t.DataInclusao,
       t.DataAlteracao,
       t.DataInativacao
   FROM Tarefa t
   WHERE t.Id = IdTarefa (quando IdTarefa não for NULL)
   LEFT JOIN quando IdTarefa for NULL (usar apenas NomeTarefa)

LÓGICA DE MAPEAMENTO E TRADUÇÃO:
1. name (TEXT, NOT NULL):
   - Se NomeTarefa da ViewOrcamentosLojas não for NULL/vazio: traduzir para inglês em snake_case
   - Se NomeTarefa for NULL/vazio: usar dbo.Tarefa.Nome (se disponível)
   - Exemplo de tradução: "REABASTECIMENTO" -> "supply_and_disruption"
   - Normalizar para snake_case (minúsculas, espaços e caracteres especiais substituídos por underscore)

2. task_type (VARCHAR(50), NOT NULL):
   - Determinar tipo baseado no nome traduzido ou usar padrão
   - Padrão: "standard" (se não for possível determinar)

3. is_active (BOOLEAN, NOT NULL):
   - Mapear dbo.Tarefa.Ativo (bit) para BOOLEAN
   - Se dbo.Tarefa não disponível: usar padrão true

4. deleted_at (TIMESTAMP WITH TIME ZONE, nullable):
   - Usar dbo.Tarefa.DataInativacao se existir
   - Caso contrário: NULL

5. created_at (TIMESTAMP WITH TIME ZONE, NOT NULL):
   - Usar dbo.Tarefa.DataInclusao se disponível
   - Caso contrário: datetime atual

6. updated_at (TIMESTAMP WITH TIME ZONE, NOT NULL):
   - Usar dbo.Tarefa.DataAlteracao se disponível
   - Caso contrário: usar created_at

ESTRATÉGIA DE UPSERT:
- Usar campo `name` como chave única para identificar duplicatas
- INSERT ... ON CONFLICT (name) DO UPDATE SET ...
- Se tarefa já existir (mesmo name): atualizar campos (task_type, is_active, deleted_at, updated_at)
- Se tarefa não existir: inserir novo registro com UUID gerado automaticamente

MAPEAMENTO PARA contract_scenarios:
Após migrar promoter_tasks, criar um mapeamento (promoter_task_map):
- Chave: (IdTarefa, NomeTarefa) ou apenas NomeTarefa normalizado
- Valor: UUID de promoter_tasks.id
- Este mapeamento será usado no step2 para preencher contract_scenarios.promoter_task_id

Query base para coleta:
SELECT DISTINCT
    v.IdTarefa,
    v.NomeTarefa
FROM ViewOrcamentosLojas v
INNER JOIN Orcamento o ON o.Id = v.IdOrcamento
WHERE v.IdOrcamento IN (valores de contracts/contracts_filter_main.json -> aggregated_ids.IdOrcamento)
    AND (v.IdTarefa IS NOT NULL OR v.NomeTarefa IS NOT NULL)
ORDER BY v.IdTarefa, v.NomeTarefa

Query base para dados complementares:
SELECT 
    t.Id,
    t.Nome,
    t.Ativo,
    t.DataInclusao,
    t.DataAlteracao,
    t.DataInativacao
FROM Tarefa t
WHERE t.Id = ? (quando IdTarefa disponível)
-- OU buscar por Nome quando IdTarefa for NULL


-- STEP 2: step2_migrate_contract_scenarios

⚠️ DEPENDÊNCIA CRÍTICA: Este step DEPENDE do step9_migrate_promoter_tasks ter sido executado.
O campo promoter_task_id (UUID) requer que promoter_tasks já esteja migrado e mapeado.

Tabela Origem: ViewOrcamentosLojas + OrcamentoLojas (LEFT JOIN)
Tabela Destino: contract_scenarios

LIMPEZA DE DADOS:
- SEM filtros aplicados no step1: TRUNCATE TABLE contract_scenarios CASCADE
- COM filtros aplicados no step1: DELETE FROM contract_scenarios WHERE contract_id IN (SELECT id FROM contracts WHERE legacy_id IN (lista de IdOrcamento))
- COM flag --clear-data: TRUNCATE TABLE contract_scenarios CASCADE

FILTRO OBRIGATÓRIO: Aplicar filtro WHERE IdOrcamento IN (lista de IdOrcamento do step1)
Fonte do filtro: contracts/contracts_filter_main.json -> aggregated_ids.IdOrcamento

Exemplo de uso do filtro:
WHERE v.IdOrcamento IN (valores de contracts/contracts_filter_main.json -> aggregated_ids.IdOrcamento)

LÓGICA DE MAPEAMENTO DE promoter_task_id (UUID):
1. Antes de inserir contract_scenarios, garantir que promoter_tasks foi migrado (step1.5)
2. Criar mapeamento promoter_task_map:
   - Chave: NomeTarefa normalizado (traduzido para inglês, snake_case) OU IdTarefa
   - Valor: UUID de promoter_tasks.id
3. Para cada registro de ViewOrcamentosLojas:
   - Se IdTarefa disponível: buscar UUID por IdTarefa (via JOIN com promoter_tasks ou mapeamento)
   - Se IdTarefa NULL mas NomeTarefa disponível: buscar UUID por NomeTarefa normalizado
   - Se nenhum disponível: ERRO (promoter_task_id é NOT NULL)
4. Preencher contract_scenarios.promoter_task_id com o UUID encontrado

Query base:
SELECT 
    v.IdOrcamentoLoja,
    v.IdOrcamento,
    v.IdEstabelecimento,
    v.NomeTarefa,
    v.Frequencia,
    v.Horas,
    v.ValorHora,
    v.DataInicioOperacao,
    v.StatusPedido,
    v.DataInclusaoOrcamentoLojas,
    v.DataAlteracaoOrcamentoLojas,
    v.IdTarefa,
    ol.Ativo
FROM ViewOrcamentosLojas v
LEFT JOIN OrcamentoLojas ol ON ol.Id = v.IdOrcamentoLoja
WHERE v.IdOrcamento IN (valores de contracts/contracts_filter_main.json -> aggregated_ids.IdOrcamento)
ORDER BY v.IdOrcamentoLoja

Query para obter UUID de promoter_task_id (após step9):
-- Opção 1: Buscar por IdTarefa (quando disponível)
SELECT pt.id
FROM pdv.promoter_tasks pt
WHERE pt.legacy_id = IdTarefa (se houver coluna legacy_id)
-- OU buscar por name normalizado

-- Opção 2: Buscar por NomeTarefa normalizado (traduzido para inglês, snake_case)
SELECT pt.id
FROM pdv.promoter_tasks pt
WHERE pt.name = NomeTarefa_normalizado

-- Opção 3: Usar mapeamento em memória (promoter_task_map) criado no step9
-- Mais eficiente: usar dicionário Python com chave (IdTarefa, NomeTarefa) -> UUID

-- RESUMO DE DEPENDÊNCIAS ENTRE STEPS --

STEP 1 (step1_migrate_contracts):
- Não depende de nenhum outro step
- Gera: contracts com legacy_id = IdOrcamento
- Gera arquivo: contracts/contracts_filter_main.json

STEP 2 (step2_migrate_contract_scenarios):
- DEPENDE DE: STEP 1 e STEP 9 (promoter_tasks)
- Filtro: WHERE IdOrcamento IN (do step1)
- Usa: promoter_task_map do step9 para preencher promoter_task_id (UUID)
- Gera: contract_scenarios com legacy_id = IdOrcamentoLoja

STEP 9 (step9_migrate_promoter_tasks):
- DEPENDE DE: STEP 1 (para filtro de IdOrcamento)
- Filtro: Coletar tarefas únicas da ViewOrcamentosLojas filtrada pelo step1
- Gera: promoter_tasks com UUID (id) e name normalizado
- Gera mapeamento: promoter_task_map (NomeTarefa/IdTarefa -> UUID)
- ⚠️ CRÍTICO: Deve ser executado ANTES do STEP 2 (na ordem de execução: step1 → step9 → step2)

STEP 3 (step3_migrate_contract_scenario_stores):
- DEPENDE DE: STEP 1 e STEP 2
- Filtro: WHERE IdOrcamentoLoja IN (do step2)
- Gera: contract_scenario_stores

STEP 4 (step4_migrate_contract_sellers):
- DEPENDE DE: STEP 1
- Filtro: WHERE Id IN (do step1) ou WHERE IdOrcamento IN (do step1)

    c.NomeFantasia,
    c.Ativo,
    c.DataInclusao,
    c.DataAlteracao,
    c.DataAtivacao
FROM Cliente c
WHERE c.Id IN (
    SELECT DISTINCT IdCliente 
    FROM ViewOrcamentosLojas
    WHERE IdCliente IS NOT NULL
)
ORDER BY c.Id

-- ARQUIVO JSON DE FILTROS (contracts_filter_main.json) --

⚠️ CRÍTICO: O arquivo JSON é gerado ANTES da migração, não depois.

REGRAS DE GERAÇÃO DO JSON:
1. Quando executado: O JSON é gerado ANTES de migrar os dados (logo após coletar IDs únicos)
2. Onde é gerado: No step1_migrate_contracts, logo após coletar IDs da ViewOrcamentosLojas
3. Por que ANTES: Para que stores e customers possam ler os filtros corretos antes de executar
4. Conteúdo do JSON:
   - filters_applied: Filtros aplicados (IdOrcamento, datas, limit_rows, clear_data)
   - aggregated_ids: IDs únicos coletados (IdOrcamento, IdCliente, IdEstabelecimento, IdBandeira, IdRede)
   - execution_info: Informações da execução (timestamp, source, total migrado)

COMPORTAMENTO:
- Se contracts é executado primeiro: Gera JSON → stores e customers leem o JSON
- Se stores/customers executam primeiro: Buscam diretamente da ViewOrcamentosLojas e atualizam o JSON
- O JSON sempre reflete os dados que SERÃO migrados, não os que JÁ FORAM migrados

ATUALIZAÇÃO DO JSON:
- Contracts: Atualiza após coletar IDs únicos (ANTES da migração)
- Stores: Atualiza quando busca diretamente da ViewOrcamentosLojas (com ou sem LIMIT)
- Customers: Atualiza quando busca diretamente da ViewOrcamentosLojas (com ou sem LIMIT)
- Flag json_updated_this_run: Garante que o JSON seja atualizado apenas uma vez por execução


-- LÓGICA DE LIMIT EM CUSTOMERS --

⚠️ IMPORTANTE: Quando há --limit em customers, o limite é aplicado APENAS aos IdCliente.

COMPORTAMENTO COM --limit:
1. Primeiro: Identificar quais IdCliente serão migrados (com TOP/LIMIT)
2. Depois: Coletar TODOS os IdEstabelecimento, IdBandeira e IdRede relacionados a esses clientes (SEM LIMIT)

EXEMPLO:
- Comando: python orchestrator_tasks.py customers --id-orcamento 6192 --limit 10
- Resultado:
  * 10 IdCliente migrados (limit aplicado)
  * TODOS os IdEstabelecimento relacionados aos 10 clientes (sem limit)
  * TODOS os IdBandeira relacionados aos 10 clientes (sem limit)
  * TODOS os IdRede relacionados aos 10 clientes (sem limit)

QUERY BASE PARA IdCliente (com LIMIT):
```sql
SELECT DISTINCT TOP 10 v.IdCliente
FROM ViewOrcamentosLojas v
INNER JOIN Orcamento o ON o.Id = v.IdOrcamento
WHERE v.IdOrcamento IN (6192)
```

QUERY PARA OUTROS IDs (SEM LIMIT):
```sql
SELECT DISTINCT v.IdEstabelecimento
FROM ViewOrcamentosLojas v
INNER JOIN Orcamento o ON o.Id = v.IdOrcamento
WHERE v.IdCliente IN (clientes_selecionados)  -- IDs dos 10 clientes selecionados
```

MOTIVAÇÃO:
- O limite deve ser aplicado apenas aos clientes (entidade principal)
- Outros IDs são coletados completamente para garantir integridade referencial


-- LÓGICA DE USERS (SEMPRE FULL QUANDO AUTOMÁTICO) --

⚠️ CRÍTICO: Users sempre executa FULL quando executado automaticamente.

REGRAS DE EXECUÇÃO:
1. Quando executado automaticamente (como dependência de contracts):
   - SEMPRE executa FULL (limit_rows=0)
   - NÃO aceita --limit mesmo se especificado
   - Motivo: Contracts depende de TODOS os users para funcionar corretamente

2. Quando executado diretamente:
   - PODE usar --limit se especificado
   - Exemplo: python orchestrator_tasks.py users --limit 10 → migra 10 linhas
   - Exemplo: python orchestrator_tasks.py users → migra todas as linhas

3. Quando executado como parte de "todas as tasks":
   - SEMPRE executa FULL (limit_rows=0)
   - NÃO usa o --limit global mesmo se especificado

CENÁRIOS:
✅ python orchestrator_tasks.py users --limit 10
   → Migra 10 linhas (execução direta, limit aplicado)

✅ python orchestrator_tasks.py users
   → Migra todas as linhas (execução direta, sem limit)

✅ python orchestrator_tasks.py contracts --id-orcamento 6192
   → Se users estiver vazio, executa users FULL automaticamente (limit_rows=0)

✅ python orchestrator_tasks.py --limit 100
   → Executa todas as tasks, mas users sempre FULL (limit_rows=0)

MOTIVAÇÃO:
- Contracts referencia users via foreign keys
- Se users estiver parcial, contracts pode falhar ao referenciar users inexistentes
- Garantir que todos os users estejam disponíveis quando contracts precisar


-- FLAG --clear-data E USERS --

⚠️ IMPORTANTE: O flag --clear-data NÃO afeta users.

REGRAS:
1. Quando --clear-data é usado em contracts:
   - Limpa apenas tabelas de contracts
   - NÃO limpa tabela users
   - Users não é afetado

2. Quando --clear-data é usado em customers:
   - Limpa apenas tabelas de customers
   - NÃO limpa tabela users
   - Users não é afetado

3. Quando --clear-data é usado em stores:
   - Limpa apenas tabelas de stores
   - NÃO limpa tabela users
   - Users não é afetado

4. Quando --clear-data é usado em users:
   - Limpa apenas tabela users
   - Comportamento normal do módulo

MOTIVAÇÃO:
- Users é uma dependência crítica de contracts
- Não podemos perder users parcialmente quando limpamos outros módulos
- Users deve ser gerenciado independentemente


-- LÓGICA DE STORES E CUSTOMERS SEMPRE USAREM ViewOrcamentosLojas --

⚠️ CRÍTICO: Stores e customers SEMPRE filtram pela ViewOrcamentosLojas, mesmo sem filtros.

REGRAS:
1. STORES:
   - Step 2 (retail_chains): SEMPRE busca IdRede da ViewOrcamentosLojas
   - Step 3 (store_brands): SEMPRE busca IdBandeira da ViewOrcamentosLojas
   - Step 4 (stores): SEMPRE busca IdEstabelecimento da ViewOrcamentosLojas
   - Mesmo em full load (sem filtros): filtra pela ViewOrcamentosLojas
   - Mesmo com --limit: primeiro busca IDs da ViewOrcamentosLojas, depois aplica limite

2. CUSTOMERS:
   - Step 2 (customers): SEMPRE busca IdCliente da ViewOrcamentosLojas
   - Mesmo em full load (sem filtros): filtra pela ViewOrcamentosLojas
   - Mesmo com --limit: primeiro busca IdCliente da ViewOrcamentosLojas, depois aplica limite

COMPORTAMENTO COM --limit:
- Com --limit 10: Busca os primeiros 10 IdCliente/IdEstabelecimento da ViewOrcamentosLojas
- Sem --limit: Busca TODOS os IdCliente/IdEstabelecimento da ViewOrcamentosLojas
- Mas SEMPRE filtra pela ViewOrcamentosLojas (nunca busca diretamente da tabela origem)

MOTIVAÇÃO:
- Apenas lojas/clientes ativos relacionados a contratos devem ser migrados
- ViewOrcamentosLojas já aplica os filtros de negócio necessários
- Garantir consistência entre contracts, stores e customers
