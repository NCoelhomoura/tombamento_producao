----------------------------------------------------------------------------------------------------
Todos os campos obrigatorios foram mapeados ou preenchidos com valores padrao.


NOTAS IMPORTANTES:
----------------------------------------------------------------------------------------------------
1. A tabela users no PostgreSQL segue um padrao de autenticacao (Identity/AspNet), entao alguns campos
   precisam ser preenchidos com valores padrao mesmo que nao existam no SQL Server.

2. O campo status deve ser mapeado de acordo com o valor de Ativo:
   - Ativo = TRUE -> status = 'active'
   - Ativo = FALSE -> status = 'inactive'

3. Os campos normalized_user_name e normalized_email devem ser preenchidos com versoes normalizadas
   (uppercase para user_name, lowercase para email).

4. O campo password_hash armazena o hash MD5 da senha do SQL Server. Se necessario, pode ser
   migrado para um formato mais seguro posteriormente.

5. O campo legacy_id foi criado manualmente para permitir o mapeamento de IdUsuarioVendedor em contracts.

6. Campos de auditoria (created_at, updated_at) devem ser preenchidos com as datas do SQL Server.

7. Campos de seguranca (email_confirmed, phone_number_confirmed, temporary_password, etc.) devem
   ser preenchidos com valores padrao apropriados.

NOTAS IMPORTANTES:
----------------------------------------------------------------------------------------------------
1. A tabela user_roles cria uma relação muitos-para-muitos entre users e roles.

2. Para cada usuário existente na tabela users (incluindo o usuário manual Sys Admin),
   será criado um registro em user_roles associando-o à role HUNTER.

3. O UUID da role HUNTER é fixo: 019bb283-9894-75e4-9c15-84491fb67224

4. A migração deve ser executada após a migração da tabela users (ETAPA 1).

5. Se a tabela user_roles já contiver registros, deve ser feita limpeza (TRUNCATE ou DELETE)
   antes de inserir os novos registros, dependendo da estratégia de migração.

6. Todos os usuários migrados receberão automaticamente a role HUNTER por padrão na primeira carga.

7. A tabela user_roles segue o padrão de auditoria com created_at e updated_at.

8. Se necessário adicionar outras roles posteriormente, isso pode ser feito manualmente ou através
   de scripts adicionais.
